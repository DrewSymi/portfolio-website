Conditional Access & MFA Policy Implementation

Risk-based authentication policies in Entra ID with break-glass accounts, exception handling, and governance documentation.

Entra ID Conditional Access MFA Risk-Based Auth Governance

Problem Statement

Organizations struggle to balance security and usability in authentication. Too strict, and productivity suffers. Too loose, and you're one phishing email away from a breach. The challenge is implementing policies that adapt to risk while maintaining operational continuity.

This project demonstrates how to design Conditional Access policies that provide strong protection without breaking legitimate workflows — including handling exceptions, break-glass scenarios, and governance requirements.

Why This Matters in IAM

Conditional Access is the enforcement point for Zero Trust in Microsoft environments. It answers the question: "Given what I know about this user, device, and context, should I allow, block, or require additional verification?"

  • Risk Reduction: Block or challenge suspicious sign-ins before damage occurs
  • Compliance: Enforce MFA requirements mandated by SOX, PCI, HIPAA
  • User Experience: Reduce friction for low-risk, compliant users
  • Audit Trail: Every policy decision is logged for investigation

Policy Architecture

Baseline MFA

Require MFA for all users accessing cloud apps from any location.

  • Target: All users
  • Condition: All cloud apps
  • Control: Require MFA

Location-Based

Block sign-ins from high-risk countries; trust corporate network locations.

  • Named locations configured
  • Block: Sanctioned countries
  • Trust: Corporate IPs

Device Compliance

Require compliant or hybrid-joined devices for sensitive apps.

  • Intune compliance required
  • Target: Finance apps
  • Control: Compliant device

Risk-Based

Respond to Microsoft's sign-in and user risk signals dynamically.

  • High risk: Block access
  • Medium risk: Require MFA
  • Password change on user risk

Break-Glass Accounts

Emergency access accounts are critical when Conditional Access or MFA fails. Without them, you could lock yourself out of your own tenant.

Break-Glass Requirements

Cloud-only accounts excluded from all Conditional Access policies. Long, complex passwords stored securely offline. Usage triggers immediate alerts. Regular testing required.

Governance & Exceptions

Policies without governance become security theater. Every exception needs documentation, justification, and expiration.

  • Exception Request Process: Documented approval workflow with business justification
  • Time-Bound Exclusions: All exceptions expire after 90 days maximum
  • Compensating Controls: If MFA is waived, what replaces it?
  • Regular Review: Monthly audit of active exceptions
  • Audit Trail: All policy changes logged with who/what/when

Risk Considerations

  • Policy Conflicts: Multiple overlapping policies can cause unexpected blocks
  • Legacy Authentication: Older protocols bypass Conditional Access — block them
  • Service Accounts: Non-human identities need different handling
  • Testing: Always use Report-Only mode before enforcing
  • Rollback Plan: Know how to quickly disable a policy if it breaks production

Lessons Learned

  • Start with Report-Only mode — never deploy blind
  • Named locations simplify policy management significantly
  • Break-glass accounts are non-negotiable
  • Document everything — you'll thank yourself during an audit
  • User communication matters — explain why MFA is required
